# Found inconsistencies

Remember: bounding and ambient caps sets are unset in `./bundle/config.json`.

- In case of `sudo ${runtime} run`:
  - we can see how `runc run`'s init explicitly drops all bounding caps and then tries to set effective/permitted/inheritable caps sets and gets EPERM because bounding is now empty.
  - On the other hand, `youki` in its init `run` path takes missing bounding set in config.json as "don't touch it", and successfully runs the container.
- In case of `sudo youki exec`:
  - we can see that youki treats missing bounding caps set the same way runc treats it in `sudo runc run`, it explicitly drops it and then gets EPERM trying to capset effective/permitted/inheritable sets because bounding is now empty.

| command | reaction to missing bounding caps | result |
|---------|-----------------------------------|--------|
| sudo runc run | explicitly drop all bounding, then capset fails | fails |
| sudo youki run | don't touch bounding | succeeds |
| sudo youki exec | explicitly drop all bounding, then capset fails | fails (at exec!) |

# Ways to fix?

1. Make youki's run path consistent with its exec path (make them both drop bounding when missing)
  - match runc, breaks any working config without explicitly set bounding capabilities (youki spec generates specs with [CAP_KILL,CAP_AUDIT_WRITE,CAP_NET_BIND_SERVICE] by default)
  - feels more secure, declarative/reproducible and conservative (I like it)
2. Make youki's exec path consistent with its run path (make them both preserve bounding when missing)
  - keep current `run` behavior, just make exec match it
  - but now it will be one of global inconsistencies with 'runc'
    that may cause security heisenbugs when someone will port their working configs from runc to youki
